[[Obsidian Publish 소개|Obsidian Publish]] 사이트에 사용자 지정 도메인 또는 하위 도메인을 설정할 수 있습니다.

> [!warning] 경고
> 현재 다음 방법을 사용한 사용자 지정 도메인 설정만 지원합니다:
>
> - [Full 모드](https://developers.cloudflare.com/ssl/origin-configuration/ssl-modes/full/)를 사용한 [[#CloudFlare를 사용한 설정]]
> - [[#프록시를 사용한 설정]]
> 
> 아직 사용자를 대신하여 SSL 인증서를 프로비저닝하는 방법은 없습니다.

## CloudFlare를 사용한 설정

사용자 지정 도메인 또는 하위 도메인을 설정하는 가장 쉬운 방법은 [CloudFlare](https://cloudflare.com) 계정을 생성하고 CloudFlare가 도메인의 DNS를 관리하도록 하는 것입니다.

다음 단계는 CloudFlare를 사용하여 Obsidian Publish 사이트에 사용자 지정 도메인을 설정하는 방법으로, 루트 도메인(`mysite.com`) 또는 하위 도메인(`notes.mysite.com`)을 사용할 수 있습니다.

> [!important] 중요
> CloudFlare는 사용자 지정 도메인 설정을 위한 **유일한 공식 지원 제공자**입니다. 다른 제공자에서 다음 지침을 사용하면 작동하지 않을 가능성이 높습니다.

**CloudFlare:**

1. Publish 사이트를 호스팅할 도메인에서 Cloudflare를 엽니다. 예를 들어 `notes.mysite.com`과 같은 하위 도메인을 원하더라도 `mysite.com`을 엽니다.
2. **DNS**로 이동하여 **Add Record**를 클릭합니다.
3. **CNAME**을 선택합니다.
4. **name**에 도메인 또는 하위 도메인을 입력합니다. 예: `notes.mysite.com`.
5. **target**에 `publish-main.obsidian.md`를 입력합니다. 이 값에 개인 하위 URL을 포함하지 마세요. Obsidian Publish가 설정에서 이를 처리합니다.
6. **proxy status**가 활성화되어 있는지 확인합니다. 기본적으로 활성화되어 있어야 합니다.
7. **SSL/TLS**로 이동하여 SSL/TLS 암호화 모드를 "Full"로 설정하면 SSL/TLS 인증서가 자동으로 구성됩니다.

> [!note] 참고
> `mysite.com`과 `www.mysite.com` 모두를 Obsidian Publish로 리디렉션하려면 다음 설정으로 [Page Rule](https://support.cloudflare.com/hc/en-us/articles/200172336-Creating-Page-Rules)을 생성해야 합니다:
>
> - URL match: `www.mysite.com/*`
> - Forward URL - 301 Permanent Redirect
> - Redirect URL: `https://mysite.com/$1`
>
> 페이지 규칙을 생성한 후 `mysite.com`에 했던 것처럼 `www.mysite.com`에 대한 CNAME 레코드를 생성합니다.

**Obsidian:**

1. 컴퓨터에서 Obsidian을 엽니다.
2. 왼쪽 [[리본]]에서 **변경 사항 게재** ( ![[lucide-send.svg#icon]] )를 클릭합니다.
3. **변경 사항 게재** 아래에서 **사이트 옵션 변경** ( ![[lucide-cog.svg#icon]] )을 선택합니다.
4. **Custom domain** 옆에서 **Configure**를 선택합니다.
5. **Custom URL**에 도메인 또는 하위 도메인의 URL을 입력합니다. 사용자 지정 URL 상자에 `www.`를 넣지 않도록 주의하세요.

> [!note] 참고
> 사용자 지정 도메인 설정이 리디렉션 루프에 빠지는 경우, CloudFlare의 암호화 모드가 "Full"이 아닌 "Flexible"로 설정되었을 가능성이 높습니다.

## 프록시를 사용한 설정

자체 웹 서버를 사용하여 사용자 지정 도메인에 대한 SSL/TLS를 설정할 수도 있습니다.

도메인 또는 하위 도메인에서 이미 웹사이트를 호스팅하고 있는 경우, 이 옵션을 사용하여 전체 사이트를 호스팅하는 대신 특정 URL 경로에서 Obsidian Publish 사이트를 로드하도록 웹사이트를 설정할 수 있습니다.

해당 URL 경로 아래의 모든 요청을 `https://publish.obsidian.md/serve?url=mysite.com/my-notes/...`로 프록시하고, Obsidian의 사이트 옵션에서 **Custom URL**을 `mysite.com/my-notes`로 설정하여 동일한 URL 경로를 구성합니다.

Obsidian Publish를 소유한 사이트의 하위 URL로 설정할 수도 있습니다. 예: `https://mysite.com/my-notes/`. 이를 위해서는 자체 서버를 호스팅하고 모든 요청을 `https://publish.obsidian.md/`의 서버로 프록시해야 합니다.

다음 프록시 설정 예시는 모든 경우를 다루지는 않지만, 이 구현을 위한 일반적인 방법을 제공합니다.

### NGINX

NGINX 설정에 다음을 추가합니다:

```nginx
location /my-notes {
  proxy_pass https://publish.obsidian.md/serve?url=mysite.com/my-notes/;
  proxy_ssl_server_name on;
  proxy_set_header Host publish.obsidian.md;
}
```


일부 사용자는 프록시 패스에 `$request_uri`를 추가해야 할 수 있다고 보고했습니다:

```nginx
location /my-notes {
  proxy_pass https://publish.obsidian.md/serve?url=mysite.com/my-notes$request_uri;
  proxy_ssl_server_name on;
  proxy_set_header Host publish.obsidian.md;
}
```

### Apache

`.htaccess`에 다음을 추가합니다:

```htaccess
RewriteEngine  on
RewriteRule    "^my-notes/(.*)$"  "https://publish.obsidian.md/serve?url=mysite.com/my-notes/$1"  [L,P]
```

> [!note] 참고
> `mod_rewrite`가 활성화되어 있어야 하며, [SSLProxyEngine](https://stackoverflow.com/questions/40938148/reverse-proxy-for-external-url-apache)을 구성해야 할 수도 있습니다.

### Netlify

`netlify.toml`에서 [리디렉션을 구성](https://docs.netlify.com/routing/redirects/#syntax-for-the-netlify-configuration-file)합니다:

```plain
[[redirects]]
  from = "https://mysite.com/my-notes/*"
  to = "https://publish.obsidian.md/serve?url=mysite.com/my-notes/:splat"
  status = 200
  force = true
```

### Vercel

`vercel.json`에서 [리라이트를 구성](https://vercel.com/docs/configuration#project/rewrites)합니다:

```json
{
  ...

  "rewrites": [
    {
      "source": "/my-notes/",
      "destination": "https://publish.obsidian.md/serve?url=mysite.com/my-notes"
    },
    {
      "source": "/my-notes/:path*",
      "destination": "https://publish.obsidian.md/serve?url=mysite.com/my-notes/:path*"
    }
  ]
}
```

### Caddy

```plain
mysite.com {
  encode zstd gzip
  handle /my-notes* {
    reverse_proxy https://publish.obsidian.md {
      header_up Host {upstream_hostport}
    }
    rewrite * /serve?url=mysite.com{path}
  }
}
```

### Traefik

이 최소 설정 발췌는 `mysite.com`을 Obsidian Publish로 리디렉션합니다.
전체 예시는 [Traefik 문서](https://doc.traefik.io/traefik/routing/overview/)를 참조하세요.

```yaml
http:
  routers:
    mysite:
      rule: Host(`mysite.com`)
      service: obsidian-publish
      middlewares:
        - "publish-headers"
  services:
    obsidian-publish:
      loadBalancer:
        servers:
          - url: https://publish.obsidian.md
  middlewares:
    publish-headers:
      headers:
        customRequestHeaders:
          Host: "publish.obsidian.md"
          x-obsidian-custom-domain: "mysite.com"
```

### 지원되는 HTTP X-Headers

프록시 서비스가 쿼리 경로를 허용하지 않는 경우, `https://publish.obsidian.md/`를 사용하고 사용자 지정 헤더 `x-obsidian-custom-domain`을 사이트 URL `mysite.com/my-subpath`로 설정할 수 있습니다.

## 이전 사이트를 사용자 지정 도메인으로 리디렉션

방문자를 이전 `publish.obsidian.md` 사이트에서 새 사용자 지정 도메인으로 리디렉션하려면 사용자 지정 도메인을 구성할 때 **사용자 지정 도메인으로 리디렉션** 옵션을 활성화하세요.

## 문제 해결

사용자 지정 도메인을 설정한 후, 이전 `https://publish.obsidian.md/slug` 링크에서 사이트를 방문한 적이 있다면 특정 기능(폰트, 그래프, 비밀번호 접근 등)이 제대로 작동하려면 브라우저 캐시를 삭제해야 할 수 있습니다. 이는 최신 브라우저에서 적용하는 교차 도메인 보안 제한 때문입니다. 좋은 소식은 사용자 지정 도메인만 사용하도록 하면 사이트 독자들은 이 문제를 겪지 않을 것이라는 점입니다.
